# AculaTrade v2.0 — Техническое задание

## Цель

Довести торговый движок до корректной работы с точностью реальных денег. Подключить Bybit для тестирования 24/7 и валидации через Veles. Заменить сырой терминал на профессиональный TradingView Advanced Charts.

## Приоритеты

1. **Корректность расчётов** — P&L, цены входа/выхода, бэктест. Ни одна цифра не должна быть неточной
2. **Профессиональный терминал** — TradingView Advanced Charts с индикаторами и drawing tools
3. **Bybit интеграция** — второй брокер, крипто 24/7, возможность сравнить с Veles
4. **Grid Trading** — сетка ордеров, основной инструмент крипто-трейдинга
5. **Валидация движка** — параллельное тестирование с Veles, совпадение результатов

## Принципы

- Ready-made first — готовые решения вместо кастомного кода
- Root-cause fixing — чиним источник проблемы, не симптом
- Тесты на каждый расчёт — 100% покрытие финансовых калькуляций
- Валидация данных — битые свечи от брокера отсеиваются до попадания в движок

## Фаза 1: Корректность расчётов

### Проблема
Стратегии показывают неточные данные, бэктест — заглушка (getSignal всегда "long"), портфель показывает бюджет из настроек (30 000₽) вместо реальных сумм операций (9 780₽). Для эмуляции трейдинга сервис непригоден.

### 1.1 Портфельная аналитика — замена кастомного кода на готовое
- Установить `@railpath/finance-toolkit` (v0.5.4, активна ноя 2025, нативный TypeScript)
- Заменить кастомный код расчёта Sharpe, Sortino, VaR, max drawdown, CAGR на вызовы из библиотеки
- Заменить кастомный расчёт корреляционной матрицы на библиотечный
- Убедиться что все метрики на странице портфеля используют библиотечные функции
- Тесты: сравнить результаты библиотеки с ручным расчётом на 3 наборах данных

### 1.2 Валидация свечей на входе
- Установить `candlestick` (v1.0.2, активна июл 2025)
- Добавить `validateOHLC` в `BrokerService.getCandles()` — каждая свеча проверяется:
  - high >= open, high >= close, high >= low
  - low <= open, low <= close
  - volume >= 0
  - timestamp валиден, не в будущем
- Битые свечи логируются и исключаются из массива
- Тесты: передать набор с 3 битыми свечами, проверить что они отфильтрованы

### 1.3 FIFO P&L калькулятор — полное покрытие тестами
- Текущий `FifoCalculator` покрыть юнит-тестами на каждый сценарий:
  - Один вход 10 лотов по 100₽, один выход 10 лотов по 110₽ → P&L = +100₽
  - Два входа (5 по 100₽, 5 по 120₽), один выход 10 по 115₽ → P&L по FIFO
  - Частичное закрытие: вход 10 лотов, выход 3 лота → остаток 7, unrealized P&L
  - Полное закрытие + новый вход в том же инструменте
  - Убыточная сделка: вход по 120₽, выход по 100₽
  - Нулевой P&L: вход = выход
- Средняя цена входа: проверить формулу на каждом сценарии
- Каждый тест с конкретными числами и ожидаемым результатом до копейки

### 1.4 Бэктест — реальная оценка условий стратегии
- Текущая проблема: `BacktestService.runBacktest()` → `getSignal` всегда возвращает `"long"`, entryConditions/exitConditions игнорируются
- Реализовать парсинг условий стратегии из `StrategyConfig.entry` и `StrategyConfig.exit`
- На каждой свече бэктеста вычислять индикаторы через `IndicatorCalculator`
- Проверять каждое условие: RSI > 70, EMA crosses above SMA, и т.д.
- Вход в позицию только когда ВСЕ entry conditions выполнены
- Выход когда ANY exit condition выполнено ИЛИ TP/SL сработал
- Тесты: простая стратегия "RSI < 30 вход, RSI > 70 выход" на известных данных → проверить точки входа/выхода

### 1.5 Портфель — реальные суммы из операций
- "Размер портфеля стратегий" = sum(amount из реальных операций BUY), не sum(strategy.risks.tradeAmount)
- Strategy card "Позиция" = initialAmount из операций (реальная сумма покупки)
- Суммарный P&L = sum(realized + unrealized по каждой стратегии)
- P&L% = суммарный P&L / суммарные вложения × 100
- Проверить что Phase 11 фиксы работают корректно на проде
- Тест: создать стратегию с бюджетом 10 000₽, реальная покупка 9 780₽ → портфель показывает 9 780₽

### Критерий завершения фазы
Все расчёты корректны до копейки. P&L совпадает при ручной проверке на калькуляторе. Бэктест отражает реальную логику стратегии. Портфель показывает реальные суммы.

## Фаза 2: TradingView Advanced Charts

### Проблема
Текущий терминал на `lightweight-charts` — только свечи, без индикаторов на графике, без drawing tools. По сравнению с Veles Finance (который использует TradingView) выглядит сырым и непрофессиональным.

### 2.1 Получение лицензии
- Подать заявку на бесплатную лицензию: https://www.tradingview.com/advanced-charts/
- Условия бесплатного использования (все выполнимы для AculaTrade):
  - Публичный сайт с бесплатной регистрацией
  - Брендинг TradingView виден на графиках (логотип/ссылка)
  - Не за paywall (доступ к чартам бесплатный)
- Получить доступ к private npm пакету

### 2.2 Замена графика
- Удалить зависимость `lightweight-charts`
- Установить TradingView Advanced Charts из private npm
- Создать компонент `TradingViewChart` взамен текущего `InstrumentChart`
- Перенести логику: realtime update последней свечи, trade markers, период-переключатель
- Убедиться что все существующие функции терминала работают (price bar, стакан, позиции, история сделок)

### 2.3 Datafeed API
- Реализовать адаптер `UDFCompatibleDatafeed` — мост между нашими данными и форматом TradingView:
  - `onReady()` — конфигурация (поддерживаемые таймфреймы, типы графиков)
  - `resolveSymbol()` — информация об инструменте (имя, тип, таймзона, точность цены)
  - `getBars()` — исторические OHLCV свечи из `BrokerService.getCandles()`
  - `subscribeBars()` — подписка на realtime обновления через наш price stream
  - `unsubscribeBars()` — отписка
- Для T-Invest: свечи через текущий `BrokerService`
- Для Bybit (фаза 3): через `BybitBrokerProvider` — тот же datafeed, другой источник
- Тесты: datafeed возвращает корректные данные для SBER (T-Invest) и BTC/USDT (Bybit mock)

### 2.4 Что получаем из коробки (не нужно реализовывать)
- 100+ технических индикаторов на графике (RSI, MACD, Bollinger, EMA, SMA, Stochastic...)
- 110+ инструментов рисования (трендлайны, фибоначчи, горизонтальные уровни, каналы...)
- 12+ типов графиков (свечи, бары, линия, Heikin Ashi, Renko...)
- Мобильная оптимизация
- Русский язык из коробки
- Сохранение настроек графика (индикаторы, рисунки) между сессиями

### Критерий завершения фазы
Терминал на уровне Veles Finance. Индикаторы добавляются в 1 клик. Drawing tools работают. Данные от обоих брокеров отображаются корректно.

## Фаза 3: Bybit Provider

### Проблема
T-Invest работает только в часы MOEX (пн-пт 10:00-23:50 МСК). Невозможно тестировать стратегии на выходных. Невозможно сравнить результаты с Veles (Veles работает только с крипто-биржами).

### 3.1 Установка и настройка SDK
- `npm i bybit-api` — готовый TypeScript SDK для Bybit v5 API (v4.6.1, активна мар 2026)
- Конфигурация: API Key + Secret через переменные окружения (`BYBIT_API_KEY`, `BYBIT_API_SECRET`)
- Поддержка testnet для безопасного тестирования без реальных денег

### 3.2 BybitBrokerProvider
- Новый класс, реализующий существующий интерфейс `BrokerProvider` (тот же паттерн что `TinkoffProvider`)
- Методы:
  - `getPortfolio()` — баланс USDT/BTC/ETH + открытые позиции с unrealized P&L
  - `getCandles(instrument, interval, from, to)` — OHLCV свечи (klines)
  - `getOrderBook(instrument, depth)` — bid/ask стакан
  - `placeOrder(instrument, side, type, quantity, price?)` — лимитный и рыночный ордер
  - `cancelOrder(orderId)` — отмена ордера
  - `getPrice(instrument)` — последняя цена
  - `getInstruments()` — список доступных торговых пар
- Маппинг типов Bybit → наши внутренние типы (Candle, Position, Order, Instrument)
- Обработка ошибок Bybit API (rate limits, insufficient balance, invalid order)
- Тесты: mock Bybit API, проверить каждый метод провайдера

### 3.3 Реалтайм стриминг цен
- WebSocket подписка на тикеры — реалтайм цены для price bar
- WebSocket подписка на orderbook — реалтайм стакан
- Интеграция с существующим price-stream-worker через Redis pub/sub
- Автореконнект при обрыве WebSocket соединения
- Тесты: mock WebSocket, проверить обработку сообщений

### 3.4 Мульти-брокер UI
- Переключатель T-Invest / Bybit в настройках пользователя (сохраняется в БД)
- При выборе Bybit:
  - Терминал показывает крипто-пары (BTC/USDT, ETH/USDT, SOL/USDT...)
  - Инструмент-селектор фильтрует по доступным парам Bybit
  - 24/7 режим — убираем проверки MOEX сессий, badge "Биржа закрыта"
  - Точность цен до 8 знаков после запятой
  - Валюта: USDT вместо ₽
- AI промпты адаптированы: учитывают что это крипто (другая волатильность, другие паттерны, 24/7 рынок)

### Критерий завершения фазы
Можно выбрать Bybit как брокера, увидеть реалтайм данные крипто в терминале, создать стратегию для BTC/USDT, получить AI анализ крипто-инструмента. Данные проходят через тот же движок что T-Invest.

## Фаза 4: Grid Trading

### Проблема
Нет сетки ордеров — основного инструмента крипто-трейдинга, который есть у Veles. Без Grid Trading невозможно провести сравнительное тестирование с Veles.

### 4.1 GridTradingService
- Алгоритм на основе OpenTrader (open-source, TypeScript, MIT лицензия)
- Логика:
  - Пользователь задаёт: инструмент, нижняя граница цены, верхняя граница, количество уровней, объём на ордер
  - Сервис рассчитывает N уровней равномерно между границами
  - На каждом уровне выставляется пара: buy limit ниже уровня, sell limit выше
  - При исполнении buy → выставляется sell на следующем уровне выше
  - При исполнении sell → выставляется buy на следующем уровне ниже
  - Бот зарабатывает на колебаниях цены внутри диапазона
- Управление жизненным циклом ордеров (pending → filled → new order)
- P&L расчёт: sum(profit от каждой пары buy-sell)
- Остановка: при выходе цены за границы диапазона — уведомление пользователю
- Тесты: симуляция 100 ценовых тиков внутри диапазона → проверить количество сделок и P&L

### 4.2 Тип стратегии GRID
- Новый тип `GRID` в `StrategyConfig`
- Параметры: `lowerPrice`, `upperPrice`, `gridLevels`, `amountPerOrder`, `instrument`
- Валидация: lower < upper, levels >= 3, amount > minimum Bybit order
- Хранение в БД через существующую модель Strategy

### 4.3 UI настройки и мониторинга
- Форма создания Grid-стратегии:
  - Выбор инструмента (из списка Bybit пар)
  - Ввод диапазона цен (от-до) с визуализацией на графике
  - Количество уровней (слайдер 3-50)
  - Объём на ордер в USDT
  - Предпросмотр: расчётная прибыль при X% волатильности
- Мониторинг активной Grid-стратегии:
  - Отображение уровней сетки на графике TradingView (горизонтальные линии)
  - Список ордеров: исполненные (зелёный), ожидающие (серый), отменённые (красный)
  - Текущий P&L сетки в реалтайме
  - Кнопка остановки: отменяет все pending ордера, фиксирует результат

### 4.4 AI интеграция
- AI анализирует волатильность инструмента за последние 30 дней
- Предлагает оптимальные параметры:
  - Диапазон на основе Bollinger Bands (±2σ от текущей цены)
  - Количество уровней на основе ATR (чем выше волатильность — больше уровней)
  - Объём на ордер на основе баланса и risk management
- Пример: "Для BTC/USDT рекомендую диапазон $62,000-$68,000, 12 уровней, $50 на ордер. Ожидаемая доходность при текущей волатильности: ~2-4% в неделю"

### Критерий завершения фазы
Grid trading работает на Bybit. Пользователь может создать сетку, видеть ордера на графике, получать P&L. AI предлагает параметры.

## Фаза 5: Валидация через Veles

### Проблема
Нет объективного подтверждения что наш движок работает корректно. Нужен эталон для сравнения.

### 5.1 Подготовка
- Зарегистрировать аккаунт на Veles Finance
- Подключить тот же Bybit аккаунт (или отдельный sub-account)
- Выбрать 2-3 инструмента для тестирования: BTC/USDT, ETH/USDT

### 5.2 Параллельный запуск
- Настроить одинаковые Grid-стратегии на AculaTrade и Veles:
  - Тот же инструмент, тот же диапазон, то же количество уровней, тот же объём
- Запустить параллельно на реальном рынке (минимальные суммы $10-50)
- Логировать каждую сделку с обеих сторон:
  - Время исполнения (с точностью до секунды)
  - Цена исполнения
  - Объём
  - Комиссия

### 5.3 Анализ и фиксы
- Сравнить по каждой стратегии:
  - Количество сделок: наши vs Veles
  - Цены исполнения: отклонение в %
  - Итоговый P&L: отклонение в %
- Для каждого расхождения:
  - Определить root cause (тайминг? округление? комиссия? условия входа?)
  - Пофиксить в нашем коде
  - Повторить тест
- Допустимое отклонение: ±1% по P&L (из-за разницы в тайминге исполнения)

### 5.4 Документирование
- Создать отчёт: `VALIDATION-REPORT.md`
- Для каждого инструмента: таблица сравнения (наши vs Veles)
- Список найденных и исправленных расхождений
- Финальное подтверждение: движок работает корректно

### Критерий завершения фазы
Результаты AculaTrade совпадают с Veles в пределах ±1%. Отчёт документирует все найденные и исправленные проблемы. Уверенность что движок работает корректно — можно применять к T-Invest.
