# Phase 02.3 Plan 02: Checker Pipeline Hardening Summary

**One-liner:** cleanTicker defense + Redis lock in signal checkAll + try/catch around lastIndicatorValues persistence in both checkers.

## What Was Built

Both `strategy-checker.ts` and `signal-checker.ts` are hardened against three failure modes identified in the audit:

1. **Dirty ticker lookup misses (D-05):** `cleanTicker()` is now applied at the start of every method that uses `strategy.instrument` or `signal.instrument` for broker/cache calls. The `@` suffix is stripped before any `getPrice`, `getCandles`, or broker lookups.

2. **Signal double-trigger via checkAll (D-09):** `SignalChecker.checkAll()` now acquires a Redis lock per signal (same pattern as `checkByInstrument`) before calling `checkSignal()`. `checkSignal()` itself no longer manages its own lock — it is a simple fetch+evaluate helper. This prevents concurrent `checkAll` cron runs from triggering the same signal twice.

3. **Indicator persistence crash (D-10):** The `lastIndicatorValues` DB update is wrapped in `try/catch` in both checkers. If Supabase is temporarily unavailable, the error is logged with context (`[StrategyChecker]` / `[SignalChecker]` prefix) but the checker continues and returns its result.
