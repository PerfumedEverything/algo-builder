# Phase 14.2: MOEX Evening Session + Chart Correctness — Research

**Researched:** 2026-03-28
**Domain:** MOEX trading sessions, T-Invest API candle fetching, price streaming, session-aware stats
**Confidence:** HIGH (all findings verified against actual codebase + installed package types)

## Summary

The terminal has a confirmed, code-verified root cause: the MOEX evening session (18:40–23:50 MSK, UTC 15:40–20:50) is silently dropped at three independent points in the system. The `candle-normalizer.ts` defaults `includeEveningSession: false` and is called with that default throughout. The `getDailySessionStatsAction` requests candles only from `10:00 MSK to now` — correct for the main session, but the to-time `today` (wall-clock) does extend into the evening; the problem is the **normalizer is not called here at all**, but the upstream `BrokerService.getCandles` feeds through `filterValidCandles` only — not `normalizeMoexCandles`. Separately, `isMarketOpen()` in `market-hours.ts` returns `false` after `18:50 MSK` (hardcoded `mskMinutes < 1130`), which may suppress live-price display in UI.

The T-Invest API (`tinkoff-invest-api` v installed) **does** return evening session candles — the `GetCandlesRequest` simply needs `from`/`to` time ranges that span 15:40–20:50 UTC. No special flag is required for historical candles via `marketdata.getCandles()`. The `CandleSource` enum confirms standard candles include all exchange candles. The streaming `lastPrice` subscription also delivers prices during the evening session without any special parameter — the stream just needs to remain connected.

The daily close discrepancy on `1d`/`1w` candles happens because MOEX officially closes the **daily** candle only after the evening session ends (~20:50 UTC / 23:50 MSK). If the app requests daily candles before that time, the last daily candle is still open and lacks the evening session volume and close price.

**Primary recommendation:** Three targeted code fixes — (1) pass `includeEveningSession: true` everywhere normalization is called or remove the normalizer filter from intraday chart candle path entirely (API already returns correct ranges), (2) extend `getDailySessionStatsAction` `from` time to `07:00 UTC` and `to` to include evening session end (`20:50 UTC`), (3) fix `isMarketOpen()` to return `true` during `15:40–20:50 UTC` (evening session).

### Actual Bug Inventory (verified in code)

#### BUG A — `candle-normalizer.ts` default drops evening session
**File:** `src/server/services/candle-normalizer.ts:32`
**Code:** `includeEveningSession: false` default in `normalizeMoexCandles`
**Used:** Called explicitly in tests and potentially in `backtest-service` / `indicator-calculator`, but **NOT** called in the main chart candle path (`BrokerService.getCandles` → `filterValidCandles` only).
**Verdict:** The normalizer is NOT the direct culprit for the chart path. The chart path does NOT call `normalizeMoexCandles`. So evening session candles likely DO appear on the chart. Need to verify whether `filterValidCandles` removes them.

#### BUG B — `getDailySessionStatsAction` session start calculation
**File:** `src/server/actions/chart-actions.ts:35-46`
**Code:**
```typescript
sessionStartUtc.setUTCHours(10 - MSK_OFFSET_MS / (60 * 60 * 1000), 0, 0, 0)
// = setUTCHours(10 - 3, ...) = setUTCHours(7, ...) = 10:00 MSK — correct
```
The `from` is correct (07:00 UTC = 10:00 MSK). The `to` is `new Date()` (current time).
**Problem:** The session stats only cover 10:00–18:40 MSK if evening candles are filtered. But since `BrokerService.getCandles` → `filterValidCandles` (no session filter), the candles returned should include 18:40–23:50 MSK if the API returns them.
**Real problem:** The stats action only calls `getDailySessionStatsAction(figi)` once on instrument select and once on period change, but does NOT periodically refresh to include new evening candles as they arrive.

#### BUG C — `isMarketOpen()` wrong boundary
**File:** `src/lib/market-hours.ts:11`
**Code:** `return mskMinutes >= 590 && mskMinutes < 1130`
- `590` minutes = 9:50 MSK (pre-open area, fine)
- `1130` minutes = 18:50 MSK — **this cuts off at 18:50, evening starts at 18:40**
- Evening session is 18:40–23:50 MSK = 1120–1430 minutes

**Impact:** `isMarketOpen()` is used in terminal for `TopMoversPanel` display context only. Does NOT block price streaming. But if other code checks this function before displaying prices, prices may appear stale.

#### BUG D — `getDailySessionStatsAction` not re-fetched during evening session
**File:** `src/app/(dashboard)/terminal/page.tsx:187-196`
`fetchDailyStats` is called:
- On instrument select (line 207)
- On period change (line 236)
- NOT on a timer

So if user opens terminal at 18:00 MSK and evening session starts at 18:40, the H/L/Vol won't update to include evening candles unless they switch instruments.

#### BUG E — Daily candle close missing evening session
**File:** `src/server/providers/broker/tinkoff-provider.ts:274-291`
`getCandles` requests `from`/`to` and does NOT specify `candleSourceType`. The default `CANDLE_SOURCE_UNSPECIFIED = 0` means "all candles." This should include evening. The daily candle `isComplete: false` during trading means the last candle is always the forming one. This is correct behavior — the close will be updated once the session ends.

## Common Pitfalls

### Pitfall 1: Assuming T-Invest API Requires a Special Flag for Evening Candles
**What goes wrong:** Developer adds `candleSourceType` or special parameter thinking it enables evening session.
**Why it happens:** Evening session seems like a special mode.
**How to avoid:** T-Invest `getCandles` returns all exchange candles within the requested time range by default (`CANDLE_SOURCE_UNSPECIFIED = 0` = all). If `from`/`to` spans evening hours, candles are returned. No extra parameter needed.
**Warning sign:** If no candles appear for 18:40–23:50, the issue is in the `from`/`to` time range or the normalizer filter, not the API parameter.

### Pitfall 2: Fixing Only the Chart, Missing the Stats Bar
**What goes wrong:** Chart shows evening candles correctly but PriceBar H/L/Vol still misses evening.
**Why it happens:** `getDailySessionStatsAction` and chart candle fetch are independent code paths.
**How to avoid:** Fix both: candle fetch path AND session stats action together in the same plan.

### Pitfall 3: `isMarketOpen()` Boundary Off-by-One
**What goes wrong:** Evening session starts 18:40 MSK = 1120 minutes. Current code uses `< 1130`. If changed to `< 1120`, it creates a 0-minute gap.
**How to avoid:** Evening session is 18:40–23:50 MSK. The correct ranges are:
- Main: `>= 600 && < 1120` (10:00–18:40 MSK)
- Evening: `>= 1120 && < 1430` (18:40–23:50 MSK)
- Combined: `(>= 600 && < 1120) || (>= 1120 && < 1430)` = `>= 600 && < 1430`

### Pitfall 4: Daily Stats Refresh Not Hooked to Evening Session
**What goes wrong:** Stats are fetched once on instrument select. Evening candles never appear in H/L/Vol/% display.
**How to avoid:** Add `setInterval(() => fetchDailyStats(instrument.figi), 60_000)` in the terminal page, active when instrument is selected and market is open (extended `isMarketOpen` including evening).

### Pitfall 5: Stale `normalizeMoexCandles` Default in Tests
**What goes wrong:** After fixing defaults, existing tests for candle-normalizer break because they test `includeEveningSession: false` explicitly but caller code changes.
**How to avoid:** The normalizer defaults should stay parametric. Change callers that need evening session to pass `{ includeEveningSession: true }`. Do NOT change the default in the normalizer itself if it's used in backtest paths where evening session may be unwanted.

### Pitfall 6: `periodOpen` in Daily/Weekly View
**What goes wrong:** For `1d`/`1w` periods, `periodOpen` is fetched from the first daily candle of the selected period. If daily candles are cached while evening session is in progress, the close is stale.
**How to avoid:** For `1d`/`1w`, the `% change` bar already fetches from `periodCandles` path in `getDailySessionStatsAction`. The issue is the real-time live price update using `usePriceStream()` for the numerator — this is already correct as the price stream is always live.

## Code Examples

### Fix 1: Correct `isMarketOpen()` — add evening session

```typescript
// src/lib/market-hours.ts
export const isMarketOpen = (now: Date = new Date()): boolean => {
  const mskOffsetMinutes = 180
  const utcMinutes = now.getUTCHours() * 60 + now.getUTCMinutes()
  const mskMinutes = (utcMinutes + mskOffsetMinutes) % 1440

  const mskDate = new Date(now.getTime() + mskOffsetMinutes * 60 * 1000)
  const mskDay = mskDate.getUTCDay()

  if (mskDay === 0 || mskDay === 6) return false

  const inMain = mskMinutes >= 600 && mskMinutes < 1120
  const inEvening = mskMinutes >= 1120 && mskMinutes < 1430
  return inMain || inEvening
}
```

### Fix 2: Periodic stats refresh in terminal page

```typescript
// src/app/(dashboard)/terminal/page.tsx
useEffect(() => {
  if (!instrument?.figi) return
  fetchDailyStats(instrument.figi)
  const id = setInterval(() => {
    if (isMarketOpen()) fetchDailyStats(instrument.figi)
  }, 60_000)
  return () => clearInterval(id)
}, [instrument, fetchDailyStats])
```

### Fix 3: Verify `candle-normalizer.ts` is NOT called in chart path

The chart path: `getCandlesForChartAction` → `BrokerService.getCandles` → `TinkoffProvider.getCandles` → `filterValidCandles`. The `filterValidCandles` is `src/server/services/candle-validator.ts`, NOT `normalizeMoexCandles`. So chart candles are NOT filtered by session — they include whatever the API returns for the time range. This means the chart should already show evening candles IF the `from`/`to` range covers them.

The only callers of `normalizeMoexCandles` to audit:
```bash
grep -rn "normalizeMoexCandles" src/
```
If called in backtest or indicator paths, keep `includeEveningSession: false` there to avoid session boundary artifacts in backtesting.

### Fix 4: T-Invest API `getCandles` — no change needed

```typescript
// tinkoff-provider.ts — current implementation is CORRECT
// GetCandlesRequest.candleSourceType defaults to CANDLE_SOURCE_UNSPECIFIED = 0 = "all candles"
// Evening session candles are exchange candles, included in UNSPECIFIED/ALL
// No change to TinkoffProvider.getCandles() needed
```

### Fix 5: `aggregateSessionStats` return type — add `periodOpen` field

Current `session-stats.ts` test at line 30 expects `{ sessionOpen, high, low, volume }` (no `periodOpen`). But `DailySessionStats` type has `periodOpen`. The existing test at line 29 does NOT test the `periodOpen` field — it would fail after any fix that returns it. Verify test expectations are consistent.

## Open Questions

1. **Does `filterValidCandles` remove evening session candles?**
   - What we know: `filterValidCandles` checks 5 OHLC invariants (from Phase 12 decision), not session times
   - What's unclear: Exact invariants — need to read `candle-validator.ts`
   - Recommendation: Read `candle-validator.ts` during planning — if it only checks OHLC sanity, evening candles pass through. If it has time checks, those must be updated.

2. **Is `normalizeMoexCandles` called anywhere in the candle fetch path for the terminal chart?**
   - What we know: The direct path does NOT call it (verified by reading `broker-service.ts` and `tinkoff-provider.ts`)
   - What's unclear: Whether `backtest-service` or `indicator-calculator` calls it with data that feeds the chart indirectly
   - Recommendation: `grep -rn "normalizeMoexCandles" src/` during planning to confirm all callers

3. **Does T-Invest API `lastPrice` stream stop during evening session?**
   - What we know: `SubscribeLastPriceRequest` has no session parameter; the stream is a persistent gRPC stream
   - What's unclear: Whether the exchange itself stops sending updates at 18:40 MSK in practice
   - Recommendation: LOW risk — T-Invest prices are documented to include evening session. The stream should continue. If the health-check in `price-stream-worker.ts` triggers (stale for >120s), it will reconnect automatically.

4. **Is there a Saturday evening session (вечерняя сессия в выходные)?**
   - What we know: `isInMoexSession` filters weekends entirely
   - What's unclear: Some MOEX instruments trade Saturday evening — likely irrelevant for stocks
   - Recommendation: Out of scope for this phase, weekend trading is edge case.
