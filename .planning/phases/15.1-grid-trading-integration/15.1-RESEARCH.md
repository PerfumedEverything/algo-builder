## Summary

Phase 15 built a working Grid Trading backend and UI, but left it as an isolated terminal panel rather than integrating it into the strategies system. This phase fixes three categories of issues: (1) a critical double P&L counting bug in `grid-trading-service.ts`, (2) several engine correctness bugs in `grid-engine.ts`, and (3) the strategy/UI integration gap so Grid strategies appear on the `/strategies` page alongside indicator strategies.

The existing code is solid in structure: `GridEngine` is a pure stateless class, `GridTradingService` handles DB coordination, `GridRepository` persists orders, `GridForm`/`GridMonitor` are complete terminal panel components. The discriminated union `StrategyConfig = IndicatorStrategyConfig | GridConfig` already exists. The `strategy` table already stores Grid strategies (timeframe='GRID', config.type='GRID'). All infrastructure is in place — no new tables or migrations are needed.

**Primary recommendation:** Fix the P&L double-counting bug first (highest impact, one-line fix), then fix engine edge cases, then wire Grid into the strategies page UI. No new npm packages needed — all dependencies already installed.

## Bug Inventory (Root Cause Analysis)

### BUG-01: CRITICAL — Double P&L counting (grid-trading-service.ts:101-110)

**Location:** `src/server/services/grid-trading-service.ts`, lines 101-110

**Root cause:** `processPriceTick` receives one `GridTickResult` per tick. That result has a single `pnlDelta` representing total profit from all fills in that tick. The loop then calls `fillOrder()` for each filled order, passing `tickResult.pnlDelta` to **every** fill. If two orders fill in one tick, P&L is recorded twice.

**Current code:**
```typescript
for (const filled of tickResult.filledOrders) {
  const ok = await this.gridRepo.fillOrder(
    gridId,
    filled.index,
    filled.side,
    currentPrice,
    tickResult.pnlDelta,  // BUG: same total P&L given to every fill
  )
  if (!ok) continue
}
```

**Fix:** `pnlDelta` in `GridTickResult` is total for the tick. Per-fill P&L must be computed at the engine level per filled SELL order (only SELLs generate realized P&L). The engine already computes per-fill P&L in the loop — it should return it per `GridLevel` (add `pnlDelta` to `GridLevel` type) OR the service should pass 0 for BUY fills and the actual per-sell profit for SELL fills.

**Simplest fix (no type change):** Since only SELL fills produce realized P&L, pass `tickResult.pnlDelta` only to the SELL fill and 0 to BUY fills. This works when at most one SELL fills per tick. For multi-fill scenarios (multiple SELLs in one tick), we need per-fill P&L from the engine.

**Correct fix (engine-level, safer):** Add `pnlDelta: number` to `GridLevel` type. Engine sets it on filled levels. Service passes `filled.pnlDelta` to `fillOrder()`. This is the correct fix.

```typescript
// GridLevel extension needed:
export type GridLevel = {
  // ...existing fields...
  pnlDelta?: number  // only set on SELL fills, 0 otherwise
}

// Engine: in processTick, when computing P&L, attach it to filled:
const filled: GridLevel = {
  ...level,
  status: 'FILLED',
  filledAt: new Date(),
  filledPrice: currentPrice,
  pnlDelta: 0,  // default, set below for SELL
}
// ...then when computing pnlDelta for SELL:
filled.pnlDelta = pnlDelta_for_this_fill
filledOrders.push(filled)

// Service:
await this.gridRepo.fillOrder(gridId, filled.index, filled.side, currentPrice, filled.pnlDelta ?? 0)
```

### BUG-02: HIGH — P&L uses SELL quantity, not BUY quantity

**Location:** `src/lib/grid-engine.ts`, lines 73-80

**Root cause:** P&L formula uses `level.quantity` (the SELL order's quantity) instead of the paired buy's quantity. SELL quantity = `amountPerOrder / sellPrice`. BUY quantity = `amountPerOrder / buyPrice`. For arithmetic grids these differ slightly. The correct formula uses the BUY quantity for the buy-cost calculation and SELL quantity for sell-revenue.

**Current code:**
```typescript
const qty = level.quantity  // BUG: this is SELL qty
const buyFee = buyPrice * qty * feeRate
const sellFee = sellPrice * qty * feeRate
pnlDelta += (sellPrice - buyPrice) * qty - buyFee - sellFee
```

**Fix:** Use `pairedBuy.quantity` for the buy side (cost + fee), `level.quantity` for sell side (revenue - fee). Since quantities may differ slightly, compute separately:
```typescript
const buyQty = pairedBuy.quantity
const sellQty = level.quantity
const buyCost = buyPrice * buyQty * (1 + feeRate)
const sellRevenue = sellPrice * sellQty * (1 - feeRate)
pnlDelta += sellRevenue - buyCost
```

### BUG-03: HIGH — Paired buy search only by index-1 (rigid)

**Location:** `src/lib/grid-engine.ts`, line 71

**Root cause:** Counter sell is placed at `level.index + 1` (one level up). When that sell later fills, the code searches for a paired buy at `level.index - 1`. This is correct for the initial cycle, but after multiple buy→sell→buy cycles, the paired buy at `index - 1` may be PENDING (not filled yet) or the actual paired buy was at a different index due to cascading counter orders.

More critically: this looks in the `levels` array passed to `processTick`, which only contains PENDING orders from DB. If the paired buy was already FILLED and removed from pending, it won't be found.

**Fix:** The service must pass ALL orders (including FILLED ones) to `processTick`, not just pending. Or the engine should track buy-fill history differently. The cleanest fix: pass `allLevels` (all orders for the grid) to `processTick` instead of only pending. The engine can then find FILLED buys across all levels.

Alternative (minimal): In `processPriceTick`, load all orders (not just pending) and pass them to the engine. Only process fills for PENDING ones, but use all for P&L pair lookup.

### BUG-04: HIGH — Counter order uses original `levels[].quantity`, not recalculated

**Location:** `src/lib/grid-engine.ts`, lines 84-100

**Root cause:** Counter order quantity comes from `levels[counterIndex].quantity` — the original initialization quantity. After a buy fills at market price, the correct counter sell quantity should be the same as the buy's quantity (sell what you bought). For exact P&L correctness, the counter sell quantity should equal the filled buy quantity.

**Current code:**
```typescript
newCounterOrders.push({
  index: counterIndex,
  price: levels[counterIndex].price,
  side: 'SELL',
  status: 'PENDING',
  quantity: levels[counterIndex].quantity,  // BUG: original init qty, not buy's qty
})
```

**Fix:** Use `level.quantity` (the buy's quantity) for the counter sell:
```typescript
newCounterOrders.push({
  index: counterIndex,
  price: levels[counterIndex].price,
  side: 'SELL',
  status: 'PENDING',
  quantity: level.quantity,  // use the filled buy's quantity
})
```

### BUG-05: MEDIUM — Price == level.price should be skipped for fill (not sell)

**Location:** `src/lib/grid-engine.ts`, lines 56-58

**Root cause:** `isSellFill = level.side === 'SELL' && currentPrice >= level.price - EPSILON` — when price exactly equals a SELL level, it fills. This is correct for paper trading (simulated fills at exact price). However, for a BUY that was just placed AS a counter order at a level, if the next tick arrives at that exact price, both BUY and SELL at that level could fill simultaneously.

**Actual issue from audit:** `price == level.price => SELL` — the concern is that a level could trigger as a SELL when it was just set as a BUY counter order. With upsert logic in `activateCounterOrder`, this should not happen in practice since the counter is a new order. This is LOW priority unless edge cases manifest.

**Fix:** For paper trading this is acceptable behavior. Skip for now or add a `createdAt` guard.

### BUG-06: MEDIUM — `calculateLevels` with `gridCount < 2` has no validation, returns [lower, upper]

**Location:** `src/lib/grid-engine.ts`, lines 11-13

**Current behavior:** Returns `[lower, upper]` silently. The schema validates `gridLevels.min(3)` at action level, but the engine itself has no guard.

**Fix:** Add `if (gridCount < 2) throw new Error('gridCount must be >= 2')` — or keep the silent fallback and add a validation note. Since schema blocks < 3 in practice, this is low risk. Keep the fallback but document it.

### BUG-07: MEDIUM — Equal lower/upper in geometric produces division by zero / NaN

**Location:** `src/lib/grid-engine.ts`, line 18

**Root cause:** `Math.pow(upperPrice / lowerPrice, i / (gridCount - 1))` — when `upper === lower`, ratio is 1, all levels are `lower`. Not NaN. When `lower === 0` (blocked by schema), division by zero. The `gridConfigSchema` has `lowerPrice: z.number().positive()` so zero is blocked at action level.

**Fix:** Add explicit validation: `if (lowerPrice <= 0) throw new Error(...)` and `if (upperPrice <= lowerPrice) throw new Error(...)` at engine level. These are redundant with schema validation but provide defense-in-depth.

### BUG-08: MEDIUM — `GridAiService.stdDev` computed but unused

**Location:** `src/server/services/grid-ai-service.ts`, line 173

**Current code:** `void stdDev` — explicitly suppresses unused variable warning. stdDev was computed for potential use in level count calculation but is not used.

**Fix:** Remove `computeStdDev` call and `stdDev` variable. Simplifies the function.

### BUG-09: MEDIUM — `expectedProfitPerGrid` doesn't account for geometric distribution

**Location:** `src/server/services/grid-ai-service.ts`, line 158

**Root cause:** Uses `step = priceRange / gridLevels` (arithmetic step) even when `gridDistribution === 'GEOMETRIC'`. For geometric, the step varies per level; the first step is smaller.

**Fix:** Since `GridAiService` always suggests `ARITHMETIC` distribution (line 180: `gridDistribution: "ARITHMETIC"`), this calculation is technically correct for its output. Document this as a known limitation. If geometric suggestion is added later, this needs updating.

### BUG-10: LOW — `stopLoss`/`takeProfit` in GridConfig ignored by processTick

**Location:** `src/server/services/grid-trading-service.ts`, `processPriceTick`

**Root cause:** `GridConfig` has optional `stopLoss` and `takeProfit` fields. `processPriceTick` never checks them.

**Fix for this phase:** Add SL/TP check after `tickResult = GridEngine.processTick(...)`. If `currentPrice <= config.stopLoss` or `currentPrice >= config.takeProfit`, call `stopGrid()`. This is a straightforward addition.

## Integration Architecture: Grid as Strategy Type

### Current State

Grid lives in terminal as a panel:
- `terminal/page.tsx` has `gridBotOpen` state, shows `GridForm` or `GridMonitor` in a side panel
- Grid strategies ARE stored in the `strategies` table (same as indicator strategies)
- The `/strategies` page renders all strategies via `StrategyCard` — but `StrategyCard` always renders `StrategyCardConditions` which assumes `config.entry`/`config.exit` (indicator config only)
- `StrategyDialog` only opens `StrategyForm` (indicator form), no Grid form path

### What Needs to Change

**1. `StrategyCard` — Grid variant rendering**

Current `StrategyCard` reads `config.entry`/`config.exit`/`config.risks` which don't exist on `GridConfig`. When a Grid strategy appears in the list, `strategy-card-conditions.tsx` will crash or show nothing.

Solution: Add a type discriminator check. `config.type === 'GRID'` renders `GridStrategyCard` (new component) or an inline Grid info block. `config.type !== 'GRID'` (or undefined) renders current indicator card.

**Pattern:**
```typescript
// In StrategyCard or strategies page:
const isGrid = (config as GridConfig).type === 'GRID'
if (isGrid) return <GridStrategyCard strategy={strategy} ... />
```

The `GridStrategyCard` shows: grid range (lower-upper), levels count, P&L from `gridStats`, fill counts, Stop button.

**2. Strategies page — "Новая стратегия" → type selector**

Currently: "Новая стратегия" opens `StrategyDialog` → AiChat + StrategyForm (indicator only).

After integration: "Новая стратегия" shows a type chooser: "Индикаторная стратегия" | "Grid Bot".
- "Индикаторная" → existing `StrategyDialog`
- "Grid Bot" → new `GridStrategyDialog` (wraps `GridForm` with instrument selection)

This can be a simple modal with two cards, or a `Select` in the existing dialog.

**3. `GridForm` needs instrument selection**

Current `GridForm` receives `instrument`/`instrumentId`/`instrumentType`/`currentPrice` as props from the terminal (where an instrument is already selected). When opened from strategies page, no instrument is pre-selected.

Solution: Add `InstrumentSelect` step to `GridForm` or create a wrapper `GridStrategyDialog` that first asks for instrument (reusing existing `InstrumentSelect` component), then shows `GridForm`.

**4. Terminal panel stays for chart overlay**

Terminal keeps `GridForm`/`GridMonitor` panel for grid creation when you're already viewing a chart. This is valuable UX. The terminal panel and strategies page are complementary entry points.

After integration: Strategies page shows created grids. Terminal panel is the creation/monitoring entry point when viewing a specific instrument.

**5. `StrategyCard` stop button for Grid**

Grid strategies need a "Stop Grid" action (calls `stopGridAction`) instead of the generic "Остановить" (calls `deactivateStrategyAction`). The stop button should appear in the card dropdown menu for Grid strategies.

## Common Pitfalls

### Pitfall 1: `StrategyCard` crashes on Grid config

**What goes wrong:** `strategy-card.tsx` line 106 reads `config.entry` and `config.exit`. Grid config has neither. TypeScript won't catch this because `StrategyConfig` is a union and the component accepts `StrategyRow` (config is `Json` from Supabase, not typed).

**How to avoid:** Add `isGridConfig(strategy.config)` guard before rendering `StrategyCardConditions`. The strategies page already passes `strategy` to `StrategyCard` — add the guard there or inside `StrategyCard`.

**Warning signs:** Runtime error "Cannot read properties of undefined (reading 'length')" when Grid strategy appears in list.

### Pitfall 2: `getOperationStatsForStrategiesAction` fails for Grid strategies

**What goes wrong:** The strategies page calls `getOperationStatsForStrategiesAction(ids, instrumentMap)` for all strategies. Grid strategies don't have standard BUY/SELL operations in the `operations` table — they have `grid_orders`. The operation stats query may return empty/incorrect data for Grid strategies.

**How to avoid:** In the strategies page, separate Grid strategies from indicator strategies. For Grid strategies, fetch stats via `getGridStatusAction` (per-grid) or add a batch grid stats action. Don't pass Grid strategy IDs to `getOperationStatsForStrategiesAction`.

### Pitfall 3: Double P&L when using old test mocks

**What goes wrong:** `grid-trading-service.test.ts` Test 3 passes `pnlDelta: 12.5` in the mock `tickResult` and then passes it to `fillOrder` mock. After the BUG-01 fix, the test must pass `filled.pnlDelta ?? 0` instead of `tickResult.pnlDelta`. Tests will fail after the fix if mock data is not updated.

**How to avoid:** Update test mocks after BUG-01 fix. Add `pnlDelta` field to `GridLevel` type and update all tests that construct `GridLevel` objects.

### Pitfall 4: `activateCounterOrder` upsert conflict creates stale orders

**What goes wrong:** `activateCounterOrder` uses upsert with `onConflict: "grid_id,level_index,side"`. If a PENDING order already exists at that (grid_id, level_index, side), it gets overwritten with new quantity/price. This is intentional but means quantity history is lost. For P&L audit purposes this is acceptable in paper trading.

**How to avoid:** Understand this is by design. Don't add extra uniqueness checks.

### Pitfall 5: GridForm rendered from strategies page without currentPrice

**What goes wrong:** `GridForm` has `currentPrice` as a required prop and uses it for default values. If opened from strategies page before instrument is selected, `currentPrice` is 0 or undefined.

**How to avoid:** The `GridStrategyDialog` must wait for instrument selection before rendering `GridForm`. Pass `currentPrice` from `InstrumentSelect.onPriceChange` callback (same pattern as `strategy-form.tsx` line 63: `const [currentPrice, setCurrentPrice] = useState<number | null>(null)`).
