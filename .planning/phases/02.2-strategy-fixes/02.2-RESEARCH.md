### Locked Decisions

**D-01:** CROSSES_ABOVE/CROSSES_BELOW — real crossing detection: trigger ONLY when indicator transitions from below→above (or above→below) the threshold. Requires storing previous indicator value per strategy/signal.

**D-02:** Add `lastIndicatorValues` JSON field to Strategy and Signal DB tables to track previous values for crossing detection.

## Summary

This phase fixes 16 bugs in the strategy/signal checker pipeline. All bugs have been located in source code and fully understood. No new libraries are needed — the fix uses existing stack (ioredis, Supabase JS SDK, technicalindicators, vitest).

The two most critical bugs are: (1) CROSSES_ABOVE/BELOW is a simple comparison instead of a crossing detector — causes continuous false-triggers every cron cycle once price crosses threshold; (2) IndicatorCalculator returns `0` for insufficient data — causes false triggers against a threshold of 0. Both bugs silently produce wrong behavior in production without any error surfacing.

The race condition bugs (triggerCount on Signal, positionState on Strategy) are partially mitigated by existing optimistic locking patterns but have gaps: the Strategy entry guard uses `.or("positionState.eq.NONE,positionState.is.null")` which breaks if a DB row ever has NULL positionState, and there is no Redis dedup between the realtime listener and the cron-based `checkAll()` running concurrently.

**Primary recommendation:** Fix all 16 bugs in four logical waves: (1) IndicatorCalculator null-safety + CROSSES detection, (2) DB atomicity + rollback, (3) Redis dedup lock + candle TTL, (4) unit tests for every fix.

### Pattern 1: Null-returning IndicatorCalculator
**What:** All indicator methods return `number | null`. Callers handle `null` as "cannot evaluate — skip this condition entirely".
**When to use:** Every indicator method call site in strategy-checker.ts and signal-checker.ts.

```typescript
// indicator-calculator.ts — AFTER fix
static calculateRSI(candles: Candle[], period = 14): number | null {
  if (candles.length < period + 1) return null
  const closes = candles.map((c) => c.close)
  const result = RSI.calculate({ values: closes, period })
  const last = result[result.length - 1]
  return last ?? null
}
```

```typescript
// Caller — evaluateCondition after fix
private evaluateCondition(condition: SignalCondition, ctx: EvalContext): boolean | null {
  const actual = this.getIndicatorValue(condition, ctx)
  if (actual === null) return null  // cannot evaluate — skip
  const target = condition.value ?? 0
  return this.compare(actual, condition.condition, target, ctx.price, condition.indicatorKey)
}
```

### Pattern 2: CROSSES Detection with lastIndicatorValues
**What:** Store `{ [indicatorKey]: previousValue }` per strategy/signal in DB. On each check: load previous, compute current, trigger if transition occurred (prev < threshold AND current >= threshold for CROSSES_ABOVE). Always save current as new previous.
**Key:** Indicator key must be unique per condition — use `{indicator}_{JSON.stringify(params)}` or positional index.

```typescript
// After crossing evaluation — update lastIndicatorValues
await this.db.from("Strategy")
  .update({ lastIndicatorValues: { ...existing, [indicatorKey]: currentValue } })
  .eq("id", strategy.id)
```

**Important:** For CROSSES detection, reading `lastIndicatorValues` from the DB row loaded at query time is sufficient — the row is already in memory from `getActiveStrategies()`.

### Pattern 3: Atomic positionState Guard (D-05, D-06)
**What:** Replace `.or("positionState.eq.NONE,positionState.is.null")` with two explicit paths.
**Current bug:** If `positionState` is NULL in DB (shouldn't happen but can on old rows), the entry guard may match unintentionally.
**Fix:** DB default ensures NONE. Code uses single `.eq()`:

```typescript
// Entry: only trigger if positionState is exactly "NONE"
query = query.eq("positionState", "NONE")
// Exit: only trigger if positionState is exactly "OPEN"
query = query.eq("positionState", "OPEN")
```

### Pattern 4: Operation Rollback (D-07)
**What:** If `recordOperation` throws after positionState was already set, roll back positionState.
**Current code gap (line 322-324):** Error is swallowed with `console.error` — state is permanently wrong.

```typescript
// AFTER fix in handleTriggered
try {
  await this.operationService.recordOperation({ ... })

  console.error(`[StrategyChecker] recordOperation failed, rolling back positionState`, e)
  await this.db.from("Strategy")
    .update({ positionState: expectedState, updatedAt: new Date().toISOString() })
    .eq("id", strategy.id)
  return

```

### Pattern 5: Redis Dedup Lock (D-12)
**What:** Before processing any strategy/signal check, acquire a Redis lock with 5s TTL using SET NX EX. If lock exists, skip.
**Lock key format:** `lock:strategy:{strategyId}` or `lock:signal:{signalId}`

```typescript
private async acquireLock(type: "strategy" | "signal", id: string): Promise<boolean> {
  const key = `lock:${type}:${id}`
  const result = await redis.set(key, "1", "EX", 5, "NX")
  return result === "OK"
}
```

### Pattern 6: CheckResult Error Flag (D-08)
**What:** Add `error?: boolean` to `CheckResult` type in both checkers.
**Current gap:** `triggered: false` is used for both "condition not met" and "system error" — impossible to distinguish in logs.

```typescript
type CheckResult = {
  strategyId: string
  strategyName: string
  instrument: string
  side: "entry" | "exit"
  triggered: boolean
```

### Pattern 7: Candle Retry (D-09)
**What:** On broker candle fetch failure, retry once after 1s before giving up.

```typescript
// Simple retry — no exponential backoff needed per decisions
try {
  candles = await broker.getCandles({ ... })
} catch {
  await new Promise((r) => setTimeout(r, 1000))
  candles = await broker.getCandles({ ... })  // throws on second fail → caught by outer try/catch
}
```

### Anti-Patterns to Avoid

- **Do not use `??0` in IndicatorCalculator return paths** — this is exactly what causes bug #7/8. Use `?? null` after fix.
- **Do not use `.or()` for positionState atomic check** — `.or()` in Supabase JS composes an OR filter on the same column but is not a DB-level atomic compare-and-swap. Use two separate `.eq()` conditions on separate code paths.
- **Do not log and continue on operation recording failure** — must rollback positionState or state becomes permanently incorrect.
- **Do not add crossing state to in-memory service** — worker restarts would lose it. Must persist in DB.

## Common Pitfalls

### Pitfall 1: CROSSES detection key collision
**What goes wrong:** Two conditions on the same indicator with different params (e.g., SMA period=20 and SMA period=50) share the same key `"SMA"` in `lastIndicatorValues`, causing them to overwrite each other's previous value.
**Why it happens:** Simple indicator name used as key.
**How to avoid:** Include params in key: `{indicator}:{JSON.stringify(sortedParams)}` or use condition array index as suffix.
**Warning signs:** Strategy with two SMA conditions of different periods behaves incorrectly.

### Pitfall 2: lastIndicatorValues not updated when condition is SKIPPED
**What goes wrong:** If a condition returns `null` (insufficient candles) and is skipped, the previous value is not updated. On next check when candles are available, the "previous" value is stale from before the gap.
**How to avoid:** Always write `currentValue` to `lastIndicatorValues` regardless of whether the condition was evaluated, as long as `currentValue !== null`.

### Pitfall 3: DB default for positionState on existing rows
**What goes wrong:** Migration adds `DEFAULT 'NONE'` for new rows, but existing rows with NULL positionState are not backfilled.
**How to avoid:** Migration must include `UPDATE "Strategy" SET "positionState" = 'NONE' WHERE "positionState" IS NULL`.

### Pitfall 4: Supabase JS `.or()` not a compare-and-swap
**What goes wrong:** `.or("positionState.eq.NONE,positionState.is.null")` is an OR filter — both conditions can be true simultaneously on a NULL row, leading to double-trigger.
**How to avoid:** Use `.eq("positionState", "NONE")` exclusively (D-05, D-06). With D-06 backfill in place, NULL rows become impossible.

### Pitfall 5: Redis lock not imported in strategy-checker / signal-checker
**What goes wrong:** `@/lib/redis` is currently only imported in `price-cache.ts`. Strategy-checker uses `PriceCache` but does not hold a direct Redis reference.
**How to avoid:** Import `redis` from `@/lib/redis` directly in strategy-checker and signal-checker for lock operations, or add lock methods to `PriceCache`.

### Pitfall 6: IndicatorCalculator return type change breaks callers
**What goes wrong:** `calculateRSI` currently returns `number`. Changing to `number | null` requires updating all call sites in both strategy-checker.ts and signal-checker.ts. Missing even one causes TypeScript errors (or runtime errors if `?? 0` fallbacks are added back).
**How to avoid:** Update the return type in IndicatorCalculator first, then let TypeScript compilation failures guide all fix locations.

### Pitfall 7: technicalindicators returns empty array (not null) on insufficient data
**What goes wrong:** `RSI.calculate({ values: [1,2], period: 14 })` returns `[]` (empty array), so `result[result.length - 1]` is `undefined`. The current code uses `?? 0`, turning undefined into 0 and silently causing false triggers.
**Root cause:** The library itself does not throw — it silently returns fewer results. Must guard BEFORE calling the library.
**How to avoid:** Check `candles.length < minimumRequired` before calling `RSI.calculate()`. Return `null` immediately.

## State of the Art

| Old Pattern | New Pattern | Why Changed |
|-------------|-------------|-------------|
| `CROSSES_ABOVE = actual > target` | `prev < target AND current >= target` | Real crossing semantics |
| `?? 0` on indicator result | `?? null` + caller skips | Prevents false triggers on empty data |

| No dedup between cron+realtime | Redis SET NX EX lock per ID | Prevents concurrent double-checks |
| `triggered: false` covers errors and non-triggers | `triggered: false, error: true` | Enables proper alerting on system errors |

## Open Questions

1. **Where was positionState column originally created?**
   - What we know: It is in `StrategyRow` type and used throughout, but not in `prisma/schema.prisma` and not in any `prisma/migrations/` file.
   - What's unclear: Whether there is a Supabase migration applied via dashboard that isn't tracked.
   - Recommendation: Migration D-06 should use `ADD COLUMN IF NOT EXISTS` to be idempotent. If column already exists with correct default, the migration is a no-op.

2. **Should lastIndicatorValues be per-condition-index or per-indicator+params key?**
   - What we know: CONTEXT.md says "JSON field to track previous values". Per-indicator+params is safer but requires stable key generation. Per-index is simpler but breaks if conditions are reordered.
   - Recommendation (Claude's Discretion): Use `{indicator}:{sorted params string}` as key. Rationale: conditions can be edited/reordered; index-based keys would silently reference wrong history.

3. **Should the Redis import be added directly to StrategyChecker/SignalChecker or routed through PriceCache?**
   - Recommendation (Claude's Discretion): Add `acquireLock` method to `PriceCache` class (it already owns the Redis connection). Keeps Redis access centralized. PriceCache is already injected into both checkers.
