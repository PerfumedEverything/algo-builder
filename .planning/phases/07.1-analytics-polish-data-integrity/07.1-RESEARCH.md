**Researched:** 2026-03-26

## Summary

Phase 7.1 replaces the Markowitz optimization UI (donut charts + rebalancing table) with actionable portfolio tools: a Health Score (1-100), plain-language diversification advice, honest benchmark comparison, human-readable correlation warnings, and a complete AI analysis prompt. The existing analytics service (`portfolio-analytics-service.ts`) already computes all the underlying data (HHI concentration, sector allocation, benchmark comparison, correlation matrix) -- this phase is primarily about **reshaping outputs** and **adding a scoring layer** on top of existing calculations.

The RF rate in the current Markowitz code is 0.16 (16%). The phase description says "RF rate = 7% (OFZ proxy)" but the actual CBR key rate as of March 20, 2026 is 15.00%, and average deposit rates are 14-15%. Using 7% would understate the risk-free comparison. Recommendation: use 15% (current CBR key rate) or make it configurable, but flag the discrepancy to the user.

**Primary recommendation:** Build a `PortfolioHealthService` with pure-function scoring methods (testable without mocks), replace MarkowitzComparison + RebalancingActions components with HealthScoreCard + DiversificationAdvice + enhanced BenchmarkCard, and rewrite the AI prompt with complete structured data.

### Pattern 1: Health Score Algorithm

**What:** Composite score 1-100 from three sub-scores: diversification (40%), risk (30%), performance (30%).

**Sub-score: Diversification (0-100)**
- HHI-based: `hhi < 0.10` = 100, `hhi < 0.15` = 80, `hhi < 0.25` = 50, `hhi >= 0.25` = 20
- Sector count bonus: `sectors >= 4` = +10, `sectors >= 6` = +20
- Dominant position penalty: any position > 40% = -30
- Final: clamp(0, 100)

**Sub-score: Risk (0-100)**
- Based on portfolio volatility from daily returns
- annualizedVol < 15% = 100, < 25% = 70, < 35% = 40, >= 35% = 20
- High-correlation penalty: each pair with |corr| > 0.7 = -5 (max -25)
- Final: clamp(0, 100)

**Sub-score: Performance (0-100)**
- Based on portfolio return vs benchmark (IMOEX)
- delta > 5% = 100, delta > 0% = 80, delta > -5% = 50, delta <= -5% = 20
- Deposit comparison: if portfolioReturn < depositRate => -20 penalty
- Final: clamp(0, 100)

**Composite:** `diversification * 0.4 + risk * 0.3 + performance * 0.3`

**Thresholds:**
- 75-100: Green (excellent) -- "Портфель в хорошей форме"
- 50-74: Yellow (needs attention) -- "Есть что улучшить"
- 0-49: Red (problems) -- "Требует внимания"

```typescript
type HealthScore = {
  total: number
  level: "excellent" | "good" | "warning" | "danger"
  diversification: { score: number; details: string[] }
  risk: { score: number; details: string[] }
  performance: { score: number; details: string[] }
}
```

### Pattern 2: Diversification Rules Engine

**What:** Replace Markowitz donuts with actionable text recommendations.

**Rules (evaluated in order, all that apply):**
1. **Single position > 30%**: "SBER 45% -- snizite do 25%, slishkom bolshaya dolya"
2. **Sector > 50%**: "Energetika 60% -- rassmotryte IT ili finansy"
3. **< 3 sectors represented**: "Tolko 2 sektora -- dobayte raznoobraziya"
4. **HHI > 0.25**: "Portfel silno skontsentrirovan"
5. **No sector X present** (where X is a major sector missing from top-3 by MOEX weight): "Net IT-sektora v portfele"
6. **All good**: "Khoroshaya diversifikatsiya"

Each rule produces an object:
```typescript
type DiversificationAdvice = {
  level: "success" | "warning" | "danger"
  icon: "check" | "alert-triangle" | "x-circle"
  text: string
}
```

### Pattern 3: Correlation-to-Text Mapper

**What:** Convert highPairs from correlation matrix into human-readable warnings.

**Logic:**
```typescript
const correlationToText = (pair: { a: string; b: string; corr: number }): string => {
  const sectorA = FUNDAMENTALS_MAP[pair.a]?.sector
  const sectorB = FUNDAMENTALS_MAP[pair.b]?.sector
  const sameSector = sectorA && sectorB && sectorA === sectorB

  if (pair.corr > 0.7) {
    const reason = sameSector ? `(oba ${SECTOR_LABELS[sectorA]})` : ""
    return `${pair.a} i ${pair.b} dvizhutsa vmeste ${reason} -- eto uvelichivaet risk`
  }
  if (pair.corr < -0.7) {
    return `${pair.a} i ${pair.b} dvizhutsa protivopolozhno -- eto khorosho dlya diversifikatsii`
  }
  return ""
}
```

### Pattern 4: Enhanced Benchmark with Deposit Comparison

**What:** Extend BenchmarkComparison type to include deposit rate comparison.

```typescript
type EnhancedBenchmarkComparison = BenchmarkComparison & {
  depositRate: number
  depositDelta: number // portfolioReturn - depositRate (annualized)
  verdict: "beats_market" | "beats_deposit" | "loses_to_deposit"
  verdictText: string
}
```

**Verdict logic:**
- `portfolioReturn > benchmarkReturn` AND `portfolioReturn > depositRate` => "beats_market": "Vy obgonyaete rynok na X%"
- `portfolioReturn > depositRate` BUT `< benchmarkReturn` => "beats_deposit": "Luchshe depozita, no rynok obgonyaet vas"
- `portfolioReturn < depositRate` => "loses_to_deposit": "Depozit vygodnee na X%"

### Pattern 5: Per-Metric Try/Catch in Server Actions

**What:** Each analytics metric wrapped individually so one failure does not break all.

export const getPortfolioAnalyticsAction = async (): Promise<ApiResponse<PortfolioAnalytics>> => {

### Anti-Patterns to Avoid
- **Keeping Markowitz Monte Carlo for display:** The 10K iteration optimizer is expensive and produces numbers meaningless to retail traders. Remove entirely from the analytics flow. Keep the code if needed for AI context only.
- **Hardcoding RF rate in service methods:** Extract to `analytics-constants.ts` for easy updates.
- **Mixing scoring logic with data fetching:** Health score computation should be a pure function taking existing analytics results as input, not re-fetching data.

## Common Pitfalls

### Pitfall 1: RF Rate Mismatch
**What goes wrong:** Phase description says "RF rate = 7% (OFZ proxy)" but current CBR key rate is 15.00% (March 2026), average deposit rate is 14-15%.
**Why it happens:** 7% may be a historical number or a simplified proxy.
**How to avoid:** Use CBR key rate (15%) for deposit comparison. For Sharpe ratio in AI context, current code already uses 0.16 which is close to reality. Store as a named constant in `analytics-constants.ts`.
**Warning signs:** "Depozit vygodnee" message showing when portfolio actually beats a realistic deposit.

### Pitfall 2: Annualization Mismatch in Benchmark Comparison
**What goes wrong:** Current getBenchmarkComparison returns period return (e.g., 5% over 90 days) but deposit rate is annual. Direct comparison is misleading.
**Why it happens:** Mixing time horizons.
**How to avoid:** Annualize portfolio return before comparing to deposit rate, OR pro-rate deposit rate to matching period: `depositRateForPeriod = annualRate * (days / 365)`.
**Warning signs:** 90-day portfolio return of 3% being compared to 15% annual deposit rate.

### Pitfall 3: Health Score Gaming by Position Count
**What goes wrong:** A portfolio with 10 tiny positions and 1 giant one scores well on sector diversity but terribly on concentration.
**Why it happens:** Counting sectors without weighting.
**How to avoid:** Use HHI (already computed) as primary diversification metric, sector count as secondary bonus only.
**Warning signs:** High sector count + high HHI.

### Pitfall 4: Empty Portfolio Edge Cases
**What goes wrong:** Health score crashes or shows misleading "0" for empty/single-position portfolios.
**Why it happens:** Division by zero, no benchmark data, no correlation possible.
**How to avoid:** Return a special "insufficient_data" state when positions < 2. Health score card shows "Nedostatochno pozitsiy dlya analiza".
**Warning signs:** NaN or Infinity in score calculations.

### Pitfall 5: Correlation Warnings Without Context
**What goes wrong:** Showing "SBER i VTBR dvizhutsya vmeste" without explaining WHY or what to do.
**Why it happens:** Just converting numbers to text without actionable advice.
**How to avoid:** Always include sector context ("oba banka") and action ("rassmotryte zamenu odnogo na instrument iz drugogo sektora").

## Code Examples

### Health Score Pure Function
```typescript
const WEIGHTS = { diversification: 0.4, risk: 0.3, performance: 0.3 }

const computeHealthScore = (
  concentration: ConcentrationIndex,
  sectorAllocation: SectorAllocation[],
  benchmarkComparison: BenchmarkComparison | null,
  highCorrelationCount: number,
  depositRate: number,
): HealthScore => {
  const divScore = computeDiversificationScore(concentration, sectorAllocation)
  const riskScore = computeRiskScore(highCorrelationCount)
  const perfScore = computePerformanceScore(benchmarkComparison, depositRate)

  const total = Math.round(
    divScore.score * WEIGHTS.diversification +
    riskScore.score * WEIGHTS.risk +
    perfScore.score * WEIGHTS.performance
  )

  const level = total >= 75 ? "excellent" : total >= 50 ? "good" : total >= 25 ? "warning" : "danger"

  return { total, level, diversification: divScore, risk: riskScore, performance: perfScore }
}
```

### Diversification Rules
```typescript
const generateDiversificationAdvice = (
  positions: PortfolioPosition[],
  concentration: ConcentrationIndex,
  sectorAllocation: SectorAllocation[],
): DiversificationAdvice[] => {
  const advice: DiversificationAdvice[] = []
  const totalValue = positions.reduce((s, p) => s + p.quantity * p.currentPrice, 0)

  for (const pos of positions) {
    const weight = (pos.quantity * pos.currentPrice) / totalValue
    if (weight > 0.3) {
      advice.push({
        level: "danger",
        icon: "alert-triangle",
        text: `${pos.ticker} ${(weight * 100).toFixed(0)}% -- snizite dolyu do 25%`,
      })
    }
  }

  for (const sector of sectorAllocation) {
    if (sector.percent > 50) {
      advice.push({
        level: "warning",
        icon: "alert-triangle",
        text: `${SECTOR_LABELS[sector.sector]} ${sector.percent.toFixed(0)}% -- slishkom mnogo`,
      })
    }
  }

  if (sectorAllocation.length < 3) {
    advice.push({
      level: "warning",
      icon: "alert-triangle",
      text: `Tolko ${sectorAllocation.length} sektora -- dobayte raznoobraziya`,
    })
  }

  if (advice.length === 0) {
    advice.push({ level: "success", icon: "check", text: "Khoroshaya diversifikatsiya" })
  }

  return advice
}
```

## Open Questions

1. **RF Rate: 7% vs 15%?**
   - What we know: Phase description says 7%, reality is 15% CBR key rate
   - What's unclear: Was 7% intentional (long-term OFZ yield proxy) or a mistake?
   - Recommendation: Use 15% for deposit comparison (what user actually gets), document as configurable constant

2. **Keep Markowitz in AI context?**
   - What we know: Markowitz output feeds into AI prompt currently
   - What's unclear: Should AI still see optimal weights or only health score?
   - Recommendation: Feed AI the health score + specific problems instead of Markowitz tables. Simpler, more actionable AI output.

3. **Portfolio volatility for risk sub-score**
   - What we know: `_buildReturnSeries` can compute daily returns for volatility
   - What's unclear: Is the extra API call overhead worth it for scoring?
   - Recommendation: Use high-correlation pair count as risk proxy (already computed, no extra calls). If portfolio volatility is wanted, reuse the returns already fetched for correlation.
