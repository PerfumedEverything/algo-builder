**Researched:** 2026-03-26

## Summary

The realtime price infrastructure is **95% built**. The price-stream-worker already subscribes to `api.stream.market.lastPrice()`, publishes to Redis pub/sub, and an SSE endpoint at `/api/prices/stream` subscribes to that Redis channel. The client-side `usePriceStream` hook connects to SSE and returns a Map of prices. The terminal page already uses this hook.

**The core problem is NOT architecture -- it is wiring.** The worker only subscribes to instruments from active strategies/signals in the DB. The terminal page needs prices for the **currently viewed instrument**, which may not be an active strategy. The worker needs a way to receive dynamic subscription requests from the frontend.

**Primary recommendation:** Add a Redis-based "requested instruments" set that the terminal writes to (via server action) and the worker reads on its refresh interval. This extends the existing architecture without adding new infrastructure. All UX fixes are straightforward component-level changes.

## Architecture Patterns

### Current Price Pipeline (WORKING)

```
price-stream-worker.ts
  -> api.stream.market.lastPrice() [gRPC stream]
  -> Redis SET price:{ticker} (cache)
  -> Redis PUBLISH price-updates (pub/sub)

/api/prices/stream (SSE route)
  -> Redis SUBSCRIBE price-updates
  -> SSE to client

usePriceStream() hook
  -> EventSource /api/prices/stream
  -> Map<instrumentId, PriceUpdate>
```

### KEY ISSUE: Worker Only Subscribes to Strategy/Signal Instruments

The worker calls `getActiveInstruments()` which queries only `Signal` and `Strategy` tables. When a user opens terminal for SBER but has no active SBER strategy, the worker does NOT stream SBER prices. The terminal falls back to 10-second API polling (`fetchApiPrice`).

### Recommended Fix: Dynamic Subscription Channel

**Pattern: Redis SET for requested instruments**

```typescript
// Terminal page (via server action) writes:
await redis.sadd("requested-instruments", ticker)
await redis.expire("requested-instruments", 300) // 5 min TTL

// Worker reads on refresh interval:
const requested = await redis.smembers("requested-instruments")
const active = await getActiveInstruments()
const all = [...new Set([...active, ...requested])]
```

This is simpler and more reliable than bidirectional Redis pub/sub communication because:
1. No new pub/sub channels needed
2. Automatic cleanup via TTL (instrument stops being requested when user leaves)
3. Worker already has a 60-second refresh loop (`refreshInstruments`)
4. Stateless -- worker restart recovers automatically

### Alternative Considered: Direct gRPC from SSE Route

Could bypass the worker and make gRPC stream calls directly from the Next.js SSE route. **Rejected** because:
- Each SSE connection would create a separate gRPC stream (expensive, rate limited)
- Worker pattern is correct -- single stream, fan-out via Redis

### Price Display Consistency

The terminal page calculates `currentPrice` as:
```typescript
const currentPrice = livePrice?.price ?? apiPrice ?? candles[last].close
```

This is correct fallback chain. The issue is that `livePrice` is rarely populated because the worker doesn't subscribe to the viewed instrument. Once dynamic subscription is working, `livePrice` will be primary source.

**For other pages (portfolio, strategies):** These use `BrokerService.getInstrumentPrice()` which calls `getLastPrices()` API (not stream). This is fine for non-realtime pages. The requirement "all price displays use same source" should mean: terminal uses streaming, other pages use cached prices (refreshed by worker into Redis, read by PriceCache).

### Change % Bug Diagnosis

Terminal calculates change% as `(currentPrice - todayOpen) / todayOpen * 100`. The `todayOpen` is fetched once via `getCandlesForChartAction` with 1d interval. Issues:

1. When switching timeframes, `todayOpen` is NOT refetched (it's only set in `handleInstrumentSelect`)
2. The `todayOpen` uses last 24h of 1d candles -- outside market hours this may return yesterday's candle
3. Fix: Always fetch todayOpen from the day candle (not dependent on selected timeframe)

## Common Pitfalls

### Pitfall 1: Worker PREFERRED_CLASS_CODES Mismatch
**What goes wrong:** Worker has `["TQBR", "TQTF", "TQOB", "TQCB", "TQIF"]` but tinkoff-provider.ts has `["TQBR", "TQTF", "TQOB", "TQCB", "TQIF", "TQGD", "TQTD"]` (TQGD/TQTD were added recently for gold tickers). Worker resolves to different instrument variant than the app.
**How to avoid:** Sync worker PREFERRED_CLASS_CODES with tinkoff-provider.ts. Extract to shared constant.

### Pitfall 2: Ticker Key Mismatch in Price Map
**What goes wrong:** Worker stores price under ticker name (e.g., "SBER"), but terminal's `usePriceStream` looks up by `instrument.ticker` or `instrument.figi`. If the worker stores by clean ticker but the UI lookup uses a different key format, prices never match.
**How to avoid:** Verify that `instrumentId` in Redis pub/sub message matches what the terminal uses for lookup. Currently worker stores ticker name, terminal looks up `prices.get(instrument.ticker)` -- these should match.

### Pitfall 3: SSE Connection Limits
**What goes wrong:** Browsers limit to 6 concurrent connections per domain. If multiple tabs open SSE, other requests may queue.
**How to avoid:** Accept this limitation (unlikely users have 6+ tabs). HTTP/2 does not have this limit -- nginx already supports it.

### Pitfall 4: Redis KEYS Command in PriceCache
**What goes wrong:** `getAllTrackedInstruments()` uses `redis.keys()` which is O(N) and blocks Redis.
**How to avoid:** Not used in hot paths currently. If needed, switch to SCAN or maintain a Set.

### Pitfall 5: Worker toNumber() Precision Mismatch
**What goes wrong:** Worker's `toNumber` does NOT use `toFixed(9)` like tinkoff-provider.ts does. This can cause kopeck-level differences.
**How to avoid:** Use the same `toFixed(9)` precision in worker.

## Code Examples

### Dynamic Instrument Subscription (Server Action)
```typescript
// src/server/actions/terminal-actions.ts
export const subscribeInstrumentAction = async (
  ticker: string,
): Promise<ApiResponse<void>> => {
  try {
    await getCurrentUserId()
    const cleanTicker = ticker.replace(/@$/, "").toUpperCase()
    await redis.sadd("requested-instruments", cleanTicker)
    await redis.expire("requested-instruments", 300)
    return successResponse(undefined)
  } catch (e) {
    return errorResponse(e instanceof Error ? e.message : "Unknown error")
  }
}
```

### Worker Reading Requested Instruments
```typescript
async function getActiveInstruments(): Promise<string[]> {
  const [signals, strategies, requested] = await Promise.all([
    db.from("Signal").select("instrument").eq("isActive", true),
    db.from("Strategy").select("instrument").eq("status", "ACTIVE"),
    redis.smembers("requested-instruments"),
  ])

  const instruments = new Set<string>()
  for (const row of signals.data ?? []) instruments.add(row.instrument)
  for (const row of strategies.data ?? []) instruments.add(row.instrument)
  for (const ticker of requested) instruments.add(ticker)
  return [...instruments]
}
```

### Date Filter Presets
```typescript
type DatePreset = "all" | "today" | "yesterday" | "7d" | "30d" | "custom"

const getDateRange = (preset: DatePreset): { from: Date; to: Date } | null => {
  const now = new Date()
  const today = new Date(now.getFullYear(), now.getMonth(), now.getDate())

  switch (preset) {
    case "all": return null
    case "today": return { from: today, to: now }
    case "yesterday": {
      const yesterday = new Date(today.getTime() - 86400000)
      return { from: yesterday, to: today }
    }
    case "7d": return { from: new Date(now.getTime() - 7 * 86400000), to: now }
    case "30d": return { from: new Date(now.getTime() - 30 * 86400000), to: now }
    case "custom": return null
  }
}
```

## Requirement-Specific Research

### RTPRICE-01: Terminal Real-time Prices (<1s delay)
**Current state:** Worker streams via gRPC -> Redis -> SSE. Pipeline exists and works for strategy instruments. Need to add dynamic subscription for terminal-viewed instruments.
**Fix:** Add `subscribeInstrumentAction`, call from terminal on instrument select, worker picks up in next refresh cycle (max 60s initial delay, then <1s streaming).
**Note:** Worker's `toNumber()` must use `toFixed(9)` to match T-Invest prices to kopeck.

### RTPRICE-02: Unified Price Source
**Current state:** Terminal uses `usePriceStream` + API polling fallback. Portfolio/strategies use `BrokerService.getInstrumentPrice()` directly.
**Recommendation:** Keep the dual approach. Terminal gets real-time via SSE. Other pages use PriceCache (populated by worker) with BrokerService fallback. The "same source" requirement means the worker is the authority -- both paths ultimately read from Tinkoff, just with different latency.

### RTPRICE-03: Paper Portfolio Date Filter
**Current state:** `PaperPortfolioView` loads all operations and has no date filtering.
**Implementation:** Add a `DatePresetSelector` component. Filter operations by date in the server action (add `from`/`to` params to `getPaperPortfolioAction`). This filters the operations used for P&L calculation, showing only trades in the selected period.

### RTPRICE-04: Hide Empty Terminal Blocks
**Current state:** PositionsPanel shows "Нет открытых позиций" message. TradeHistoryPanel shows "Нет сделок" message.
**Fix:** Conditionally render entire panel only when loading or has data. Simple JSX conditional.

### RTPRICE-05: TopMovers Clickable Links
**Current state:** TopMovers items are already `<button>` elements with `onClick={() => onSelect(mover.ticker)}`. The `onSelect` calls `handleQuickSelect` which calls `findInstrumentByTickerAction` then `handleInstrumentSelect`.
**Issue:** This already works! The requirement may be about making them look like links (underline, link cursor) rather than buttons. Or about navigation to a dedicated instrument page. Verify: the current implementation DOES navigate the terminal chart. The UX fix may just be visual styling.

### RTPRICE-06: Mobile AI Button Full Label
**Current state:** `<span className="sm:hidden">ИИ</span>` -- shows only "ИИ" on mobile.
**Fix:** Change to full label text on mobile.

### RTPRICE-07: Actual Purchase Amounts
**Current state:** `OperationService.recordOperation()` calculates `actualAmount = quantity * price` where `quantity = Math.floor(tradeAmount / price)`. The `tradeAmount` comes from strategy settings (default 10,000). The `stats.initialAmount` is sum of all buy amounts (actual, not settings).
**Issue:** The paper portfolio table does NOT show individual operation amounts. It shows total P&L. The requirement is likely about showing the actual amount spent per strategy (from operations) vs the configured `tradeAmount` in strategy settings. Need to verify what "actual purchase amounts" means in context -- it may mean showing `stats.initialAmount` (sum of actual buys) rather than `strategy.tradeAmount` (configured setting).

### RTPRICE-08: Uppercase Tickers
**Current state:** `cleanTicker().toUpperCase()` already added to strategy/signal actions. Worker uses `ticker.toUpperCase()` in resolution.
**Fix:** Audit remaining places: paper portfolio display, TopMovers, terminal price bar. Ensure all displayed tickers are uppercase.

## Architecture Decision: Worker vs In-Process Stream

**Decision: Keep the worker (price-stream-worker).**

Reasons:
1. Worker is already deployed as a separate Docker container (`Dockerfile.worker`)
2. gRPC streams are long-lived -- they don't belong in serverless-friendly Next.js API routes
3. Worker handles reconnection, health checks, candle refresh
4. Single stream per instrument (not per-user), fan out via Redis

The worker architecture is correct. Only needs:
1. Dynamic subscription support (Redis set)
2. toNumber precision fix
3. PREFERRED_CLASS_CODES sync

## Open Questions

1. **TopMovers "clickable links" -- what exactly?**
   - What we know: TopMovers items already navigate the terminal chart on click
   - What's unclear: Does Anton want URL links (like `<a href="/terminal?ticker=SBER">`) or just visual styling as links?
   - Recommendation: Treat as visual styling -- items should look more like links. Current implementation already handles navigation correctly.

2. **"Actual purchase amounts" -- exact meaning?**
   - What we know: OperationService stores actual amounts (quantity * price), not strategy settings amount
   - What's unclear: Where should this be displayed? Paper portfolio currently shows total P&L, not individual amounts
   - Recommendation: Show `stats.initialAmount` (total invested) in paper portfolio table. This IS the actual purchase amount (sum of buys), not the strategy's configured `tradeAmount`.

3. **Worker refresh delay for dynamic subscriptions**
   - What we know: Worker refreshes instruments every 60s. New terminal subscription may wait up to 60s.
   - What's unclear: Is 60s acceptable for first price update?
   - Recommendation: Reduce to 10s for the requested-instruments check only, keep 60s for DB queries. Or: on first instrument select, also do API polling until streaming kicks in (current behavior).
