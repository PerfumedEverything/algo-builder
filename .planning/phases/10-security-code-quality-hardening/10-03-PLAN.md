    **SEC-08: MIME-based extension in admin-actions.ts**
    - Near the top of the file (after imports), add a MIME-to-extension map:
      ```
      const MIME_TO_EXT: Record<string, string> = {
        "image/jpeg": "jpg",
        "image/png": "png",
        "image/webp": "webp",
      }
      ```
    - In `uploadBrokerLogoAction`, replace line 147:
      `const ext = file.name.split(".").pop() ?? "png"`
      with:
      `const ext = MIME_TO_EXT[file.type] ?? "png"`
    - This derives the extension from the validated MIME type (already checked by ALLOWED_TYPES on line ~138), not the user-supplied filename which can be spoofed.

    **SEC-10: Docker Redis hardening in docker-compose.yml**
    - In the `redis` service section, add a `command` field:
      `command: redis-server --requirepass ${REDIS_PASSWORD} --maxmemory 256mb --maxmemory-policy allkeys-lru`
    - The `REDIS_PASSWORD` env var is read from `.env` via the existing `env_file` pattern used by other services. However, the redis service does NOT have `env_file` — Docker Compose variable substitution reads from the host `.env` file in the compose directory, so `${REDIS_PASSWORD}` will be interpolated from the `.env` file at compose-up time.
    - IMPORTANT: Do NOT add `env_file: .env` to the redis service — the redis-server command receives the password as a CLI arg, not as an env var inside the container.
    - Note: The REDIS_URL in .env for other services will need to be updated to include the password (format: `redis://:password@redis:6379`). Add a comment above the redis service: `# REDIS_URL must include password: redis://:${REDIS_PASSWORD}@redis:6379`
  </action>
