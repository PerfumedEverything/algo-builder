## Summary

Phase 02.1 replaces the existing lightweight-charts terminal with a professional trading layout. The core complexity areas are: (1) TradingView widget embed and dynamic symbol switching in React/Next.js, (2) realtime price delivery from Redis pub/sub to the browser via SSE, (3) T-Invest getOrderBook REST call (API exists, verified in SDK), and (4) MOEX ISS top movers endpoint.

The price-stream-worker already publishes to the Redis `price-updates` channel keyed by ticker. The missing piece is an SSE API route that subscribes to that channel and streams events to the browser. All other data sources are verified to exist in the codebase.

The TradingView Advanced Chart widget is a `<script>` injection pattern — NOT an iframe src. Dynamically changing the symbol requires remounting the container (key trick or full DOM replace) because the widget does not expose a postMessage API for symbol changes.

**Primary recommendation:** Use SSE route at `/api/prices/stream` backed by Redis subscriber + polling fallback. Remount TradingView widget on symbol change via React `key` prop. Fetch order book via new server action wrapping `api.marketdata.getOrderBook()`.

### Pattern 1: TradingView Widget Remount
**What:** TradingView widget is embedded via a `<script>` tag that reads a JSON config block. It does not expose a JavaScript API for symbol changes after mount. The only reliable way to change symbol is to destroy and recreate the DOM container.

**When to use:** Every time the user selects a different instrument.

**Example:**
```typescript
// Source: TradingView widget-docs + community pattern
// tradingview-widget.tsx
"use client"
import { useEffect, useRef } from "react"

type Props = {
  symbol: string   // e.g. "MOEX:SBER"
  theme: "light" | "dark"
}

export function TradingViewWidget({ symbol, theme }: Props) {
  const containerRef = useRef<HTMLDivElement>(null)

  useEffect(() => {
    const container = containerRef.current
    if (!container) return

    // Clear previous widget
    container.innerHTML = ""

    const script = document.createElement("script")
    script.src = "https://s3.tradingview.com/external-embedding/embed-widget-advanced-chart.js"
    script.type = "text/javascript"
    script.async = true
    script.innerHTML = JSON.stringify({
      symbol,
      interval: "D",
      timezone: "Europe/Moscow",
      theme,
      style: "1",
      locale: "ru",
      allow_symbol_change: false,
      calendar: false,
      hide_top_toolbar: false,
      support_host: "https://www.tradingview.com",
      width: "100%",
      height: "100%",
    })

    container.appendChild(script)

    return () => {
      container.innerHTML = ""
    }
  }, [symbol, theme]) // re-runs on symbol change — key insight

  return (
    <div ref={containerRef} className="tradingview-widget-container h-full w-full" />
  )
}
```

**MOEX symbol format:** `MOEX:SBER`, `MOEX:GAZP`, `MOEX:LKOH` — confirmed working on TradingView.

**Alternative approach (simpler):** Use React `key` prop on the component. When ticker changes, React unmounts and remounts the component entirely, triggering `useEffect` cleanup + fresh mount. This is cleaner than manual DOM manipulation.

```typescript
// In parent page component:
<TradingViewWidget key={instrument?.ticker} symbol={`MOEX:${instrument?.ticker}`} theme="dark" />
```

### Pattern 2: SSE Route for Realtime Prices
**What:** Next.js Route Handler that subscribes to Redis `price-updates` pub/sub channel and streams events via Server-Sent Events to the browser.

**When to use:** Price bar (top) and order book refresh trigger.

**Example:**
```typescript
// Source: Next.js Route Handlers docs + ioredis pub/sub pattern
// src/app/api/prices/stream/route.ts
import Redis from "ioredis"
import { createClient } from "@/lib/supabase-server"
import { getCurrentUserId } from "@/server/actions/helpers"

export async function GET(request: Request) {
  // Auth check
  // ...

  const subscriber = new Redis(process.env.REDIS_URL!)
  const encoder = new TextEncoder()

  const stream = new ReadableStream({
    start(controller) {
      subscriber.subscribe("price-updates")
      subscriber.on("message", (_channel, message) => {
        controller.enqueue(encoder.encode(`data: ${message}\n\n`))
      })
    },
    cancel() {
      subscriber.unsubscribe()
      subscriber.disconnect()
    },
  })

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
      "Connection": "keep-alive",
    },
  })
}
```

**Client hook:**
```typescript
// src/hooks/use-price-stream.ts
"use client"
import { useEffect, useState } from "react"

export function usePriceStream(ticker: string) {
  const [price, setPrice] = useState<number | null>(null)

  useEffect(() => {
    if (!ticker) return
    const es = new EventSource("/api/prices/stream")
    es.onmessage = (e) => {
      const data = JSON.parse(e.data) as { instrumentId: string; price: number }
      if (data.instrumentId === ticker) setPrice(data.price)
    }
    return () => es.close()
  }, [ticker])

  return price
}
```

**SSE vs polling decision:** Use SSE as primary. The price-stream-worker already publishes to `price-updates` on every T-Invest WebSocket tick. SSE delivers sub-second latency. Polling at 1s interval is acceptable fallback if SSE connection management proves problematic with Docker/nginx reverse proxy (need `X-Accel-Buffering: no` header for nginx).

### Pattern 3: T-Invest Order Book Fetch
**What:** REST call to `api.marketdata.getOrderBook()` — NOT a streaming subscription. Called on ticker change and periodically (every 5s) for live updates.

**Example:**
```typescript
// Verified in tinkoff-invest-api SDK v7.0.1 types
// GetOrderBookRequest: { instrumentId: string, depth: number }
// GetOrderBookResponse: { bids: Order[], asks: Order[], ... }
// Order: { price: Quotation, quantity: number }

// src/server/actions/terminal-actions.ts
export const getOrderBookAction = async (
  figi: string,
  depth = 10,
): Promise<ApiResponse<OrderBookData>> => {
  const userId = await getCurrentUserId()
  const broker = await getBrokerForUser(userId)
  const result = await broker.api.marketdata.getOrderBook({
    instrumentId: figi,
    depth,
  })
  // Map Quotation to number: units + nano/1e9
  const mapOrders = (orders: Order[]) =>
    orders.map((o) => ({
      price: toNumber(o.price),
      quantity: o.quantity,
    }))
  return successResponse({
    bids: mapOrders(result.bids),
    asks: mapOrders(result.asks),
  })
}
```

**Depth recommendation:** 10 levels per side (configurable, Claude's discretion).

**Important:** `getOrderBook()` uses the system token (from TinkoffProvider's `api`) or user token. The existing `TinkoffProvider` accesses `this.api` which requires a user connection. However, order book is public data — consider whether to use `TINKOFF_SYSTEM_TOKEN` directly (as price-stream-worker does) to avoid requiring user broker connection for this read-only endpoint.

### Pattern 4: MOEX ISS Top Movers
**What:** Fetch `https://iss.moex.com/iss/engines/stock/markets/shares/boards/TQBR/securities.json?iss.meta=off` → sort by `LASTCHANGEPRCNT` → take top-N positive (gainers) and top-N negative (losers).

**Cache strategy:** Redis with 60s TTL — data is updated by exchange every minute.

**Example:**
```typescript
// Source: MOEXProvider pattern (existing in src/server/providers/analytics/moex-provider.ts)
// Extend MOEXProvider with getTopMovers()

async getTopMovers(topN = 5): Promise<{ gainers: TopMover[], losers: TopMover[] }> {
  const cacheKey = "moex:top-movers"
  const cached = await redis.get(cacheKey)
  if (cached) return JSON.parse(cached)

  const url = "https://iss.moex.com/iss/engines/stock/markets/shares/boards/TQBR/securities.json?iss.meta=off&marketdata.columns=SECID,LAST,LASTCHANGEPRCNT,VOLTODAY,HIGH,LOW"
  const res = await fetch(url)
  if (!res.ok) throw new Error(`MOEX ISS error: ${res.status}`)
  const json = await res.json()

  const { columns, data } = json.marketdata
  const rows = (data as unknown[][]).map((row) =>
    mapColumnsToObject<TopMoverRaw>(columns, row)
  ).filter((r) => r.LAST !== null && r.LASTCHANGEPRCNT !== null)

  rows.sort((a, b) => b.LASTCHANGEPRCNT - a.LASTCHANGEPRCNT)

  const gainers = rows.slice(0, topN).map(mapTopMover)
  const losers = rows.slice(-topN).reverse().map(mapTopMover)

  await redis.set(cacheKey, JSON.stringify({ gainers, losers }), "EX", 60)
  return { gainers, losers }
}
```

**Note on MOEX ISS endpoint:** `securities.json` returns both `securities` and `marketdata` blocks. The `marketdata` block contains `LAST`, `LASTCHANGEPRCNT`, `VOLTODAY`, `HIGH`, `LOW`. The `securities` block contains `SECID`, `SHORTNAME`, `LOTSIZE`. Join on `SECID`.

### Pattern 5: Positions Panel with Realtime P&L
**What:** Reuse `getPortfolioAction()` to load positions. Subscribe to SSE price stream. For each position, calculate unrealized P&L live: `(currentPrice - avgPrice) * quantity`.

**When to use:** "Open positions" panel below the chart.

**Key detail:** `getPortfolioAction` fetches from T-Invest API (slow, ~1-3s). Load once on page mount. Price updates come from SSE — apply them locally in React state without re-fetching the full portfolio.

### Anti-Patterns to Avoid
- **Trying to postMessage the TradingView widget:** The free Advanced Chart widget does not expose a JS API. Only the paid Charting Library does. Remounting is the correct approach.
- **Creating a new Redis connection per SSE request:** Use `new Redis()` per SSE connection (each needs its own subscriber), but disconnect in the `cancel` callback to avoid leaks.
- **Calling getPortfolioAction() on every price tick:** Fetch portfolio once, update prices locally.
- **Using `api.marketdata.getOrderBook()` via user token for a public endpoint:** Consider system token to avoid dependency on user broker connection status.
- **Fetching top movers without caching:** MOEX ISS returns ~200+ rows; cache with 60s TTL to avoid hammering the API.

## Common Pitfalls

### Pitfall 1: TradingView Widget Script Tag in React
**What goes wrong:** React renders the component, the script tag is injected, but on re-render React doesn't re-inject the script (React deduplicates script tags). The widget never reloads on symbol change.
**Why it happens:** React's reconciliation keeps the existing DOM node if the component doesn't unmount.
**How to avoid:** Use `key` prop to force full unmount/remount on symbol change. Or use `useEffect` with manual DOM manipulation (clear innerHTML → append new script).
**Warning signs:** Chart shows correct symbol on first load but stays the same after ticker change.

### Pitfall 2: SSE Connection Leaks in Next.js Route Handler
**What goes wrong:** Redis subscriber connections accumulate; server runs out of connections.
**Why it happens:** ReadableStream `cancel()` is not called when client disconnects in some proxy configurations.
**How to avoid:** Listen to `request.signal.aborted` in addition to `cancel()`:
```typescript
request.signal.addEventListener("abort", () => {
  subscriber.unsubscribe()
  subscriber.disconnect()
})
```
**Warning signs:** Growing Redis connection count over time.

### Pitfall 3: Nginx Buffering Breaks SSE
**What goes wrong:** SSE events arrive at browser in batches rather than individually, or not at all.
**Why it happens:** Nginx buffers proxy responses by default.
**How to avoid:** Add `X-Accel-Buffering: no` response header in the SSE route. The project deploys with Docker + nginx at aculatrade.com.
**Warning signs:** Events arrive in bursts on prod but work fine in development.

### Pitfall 4: Order Book Uses User Token (Broker Not Connected)
**What goes wrong:** `getOrderBookAction` fails for users who haven't connected their broker.
**Why it happens:** `TinkoffProvider` requires user API token. Order book is public exchange data.
**How to avoid:** Use `TINKOFF_SYSTEM_TOKEN` (already used in price-stream-worker) for order book fetches, not the user's personal token. Create a system-level TinkoffInvestApi instance in the server action.
**Warning signs:** 403/auth errors on order book for non-connected users.

### Pitfall 5: MOEX ISS `securities` vs `marketdata` Blocks
**What goes wrong:** Attempting to read `LASTCHANGEPRCNT` from the `securities` block — it's not there.
**Why it happens:** The `securities.json` endpoint returns two tables: `securities` (static data: name, lot) and `marketdata` (live data: price, change %). They must be joined on `SECID`.
**How to avoid:** Always read `LASTCHANGEPRCNT`, `LAST`, `VOLTODAY` from the `marketdata` block. Read `SHORTNAME` from the `securities` block. Join on array index (same order guaranteed) or on `SECID` column.
**Warning signs:** `LASTCHANGEPRCNT` is `null` for all rows when reading from wrong block.

### Pitfall 6: TradingView Widget `allow_symbol_change: true` Conflicts with App Selector
**What goes wrong:** User changes symbol in the TradingView widget toolbar directly — the app's InstrumentSelect component doesn't update, causing state mismatch.
**Why it happens:** Widget has its own symbol picker when `allow_symbol_change: true`.
**How to avoid:** Set `allow_symbol_change: false` in widget config. Symbol selection is exclusively via the app's InstrumentSelect.

### SSE Route with abort signal cleanup
```typescript
// src/app/api/prices/stream/route.ts
export async function GET(request: Request) {
  const subscriber = new Redis(process.env.REDIS_URL!)
  const encoder = new TextEncoder()

  const stream = new ReadableStream({
    async start(controller) {
      await subscriber.subscribe("price-updates")
      subscriber.on("message", (_ch, msg) => {
        controller.enqueue(encoder.encode(`data: ${msg}\n\n`))
      })
      request.signal.addEventListener("abort", () => {
        subscriber.unsubscribe().then(() => subscriber.disconnect())
      })
    },
    cancel() {
      subscriber.unsubscribe().then(() => subscriber.disconnect())
    },
  })

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
      "Connection": "keep-alive",
      "X-Accel-Buffering": "no",  // Critical for nginx prod
    },
  })
}
```

## Open Questions

1. **Order book token: system vs user token**
   - What we know: `getOrderBook()` is a public REST endpoint (exchange data, no account required). `TinkoffProvider` uses user token. price-stream-worker uses `TINKOFF_SYSTEM_TOKEN`.
   - What's unclear: Whether the system token account has market data subscription that covers order book in production vs sandbox.
   - Recommendation: Implement `getOrderBookAction` with system token (same pattern as price-stream-worker). If system token lacks permission, fall back to user token with graceful degradation.

2. **Price bar bid/ask source**
   - What we know: D-07/D-08 say price bar should show bid/ask live. Current `price-updates` Redis channel only publishes `lastPrice` (executed price).
   - What's unclear: Whether bid/ask from order book top-of-book should be included in price bar, or if best bid/ask comes from a separate subscription.
   - Recommendation: Fetch bid/ask from `getOrderBook(depth=1)` on the same poll interval (5s) as the full order book. Best bid = `bids[0].price`, best ask = `asks[0].price`. Display in price bar from order book response, not from SSE stream.

3. **SSE auth: how to authenticate the stream route**
   - What we know: SSE uses GET request. Server Actions use cookies for auth. Route Handlers can also read cookies.
   - What's unclear: Current `getCurrentUserId()` helper works in Server Actions — verify it works in Route Handlers too (it uses `cookies()` from `next/headers`, which is available in Route Handlers).
   - Recommendation: Call `getCurrentUserId()` at the start of the SSE GET handler. Return 401 if not authenticated.
